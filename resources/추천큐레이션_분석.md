# 추천큐레이션 이미지 분석

## 이미지 개요

- 원본 이미지: `추천큐레이션.png`
- 화면 성격: 어드민 큐레이션 작성 화면과 모바일 사용자 노출 화면을 함께 보여주는 시안
- 큐레이션 유형: 위스키 여러 개를 추천 목록으로 구성하는 카드형 큐레이션
- 핵심 패턴: 기본 발행 정보 + 위스키 N개 + 각 위스키별 큐레이터 코멘트

## 화면에서 확인되는 구성

### 1. 기본정보 영역

어드민 입력 화면의 첫 번째 섹션은 공통 발행 메타데이터를 입력하는 영역이다.

- 큐레이션명 필수 입력
- 큐레이션내용 필수 입력
- 이미지 업로드 필수
- 노출기간 시작일 필수
- 노출기간 종료일 필수
- 노출순서 입력
- 활성화 여부 토글

이미지 업로드 영역은 드래그 앤 드롭을 전제로 하며, `PNG`, `JPG`, `WEBP` 형식을 안내한다. 모바일 출력에서는 이 이미지가 상단 커버 또는 배경 영역으로 쓰이는 구조로 보인다.

### 2. 위스키 목록 영역

두 번째 섹션은 추천 대상 위스키 목록이다.

- 섹션 제목: `위스키`
- 반복 단위: `위스키 1`, `위스키 2`, ...
- 각 반복 단위는 위스키 검색/선택과 코멘트 입력을 포함한다.
- 선택된 위스키는 카드 형태로 표시된다.
- 카드에는 이미지, 한글명, 영문명, 기본 정보, 태그가 함께 표시된다.
- 항목 삭제 버튼이 있다.
- 항목 추가 버튼이 있다.

첫 번째 항목은 이미 선택된 위스키 카드로 보인다. 두 번째 항목은 검색 입력창이 비어 있고, 아래에 코멘트 입력 영역이 있다.

## 모바일 출력에서 확인되는 구조

모바일 미리보기는 사용자 앱 상세 화면 형태다.

- 상단 앱 바
- 커버 이미지 또는 그라데이션 배경
- 큐레이션 유형 배지
- 큐레이션 제목
- 노출 기간 또는 작성일로 보이는 짧은 날짜 텍스트
- 큐레이션 설명
- `위스키 큐레이션` 섹션
- 추천 위스키 카드 목록

각 위스키 카드는 다음 정보를 포함한다.

- 병 이미지
- 한글명
- 영문명
- 도수, 용량, 카테고리 등 요약 정보
- 평점 또는 평가 관련 지표
- 테이스팅 태그 칩 목록
- 큐레이터 코멘트 본문

## 출력해야 할 것

이 유형의 사용자 출력은 단순히 저장 payload를 그대로 보여주면 안 된다. 저장 payload에는 `alcoholId`와 코멘트만 있더라도, 출력 응답은 alcohol 상세 정보가 hydrate된 카드 목록이어야 한다.

필수 출력 후보:

- 큐레이션 id
- 큐레이션 유형
- 큐레이션명
- 큐레이션내용
- 커버 이미지 URL
- 노출 시작일
- 노출 종료일
- 노출순서
- 활성화 여부
- 추천 위스키 목록
- 각 위스키의 정렬 순서
- 각 위스키의 alcohol id
- 각 위스키의 표시용 alcohol 상세 정보
- 각 위스키별 큐레이터 코멘트

위스키 카드 출력 후보:

- alcoholId
- korName
- engName
- imageUrl
- category
- abv
- volume
- region 또는 country
- cask
- averageRating
- ratingCount
- reviewCount
- pickCount
- tastingTags

## 주의해야 할 것

### 1. 목록 순서 보존

추천 큐레이션은 순서가 의미를 가진다. 저장 구조가 Set 기반이면 순서가 깨질 수 있으므로, item 단위의 `displayOrder` 또는 배열 순서 보존이 필요하다.

### 2. 위스키 카드 정보는 snapshot인지 live hydrate인지 결정 필요

출력 카드의 alcohol 정보는 현재 마스터 데이터를 hydrate한 값인지, 큐레이션 발행 시점의 snapshot인지 결정해야 한다.

- live hydrate: alcohol 정보 변경 시 큐레이션 출력도 자동 변경됨
- snapshot: 발행 당시 의도를 보존하지만 저장 데이터가 커짐

운영 큐레이션이라면 최소한 alcoholId는 보존하고, 이름/이미지 변경 영향 범위를 명확히 해야 한다.

### 3. 코멘트는 큐레이션 고유 데이터

위스키의 리뷰나 설명과 다르게, 큐레이터 코멘트는 이 큐레이션 항목에 종속된다. alcohol 도메인에 넣으면 안 되고 curation payload 또는 curation item 쪽에 남아야 한다.

### 4. 비활성/삭제된 위스키 처리

선택 당시에는 존재하던 위스키가 이후 삭제되거나 비활성화될 수 있다. 상세 출력에서 항목을 숨길지, 비활성 표시를 할지, 큐레이션 자체를 오류로 볼지 정책이 필요하다.

### 5. 태그와 지표의 비용

모바일 카드에 태그, 평점, 리뷰 수, 찜 수가 모두 노출된다. 목록형 큐레이션에서 N개 위스키를 hydrate할 때 N+1 조회가 생기기 쉬우므로 batch 조회 또는 전용 projection이 필요하다.

### 6. 이미지 업로드와 커버 출력

어드민 입력의 이미지 업로드는 모바일 상단 커버에 직접 영향을 준다. 이미지가 없는 경우의 fallback, 어두운 배경 위 제목 가독성, 비율 crop 정책이 필요하다.

### 7. 노출기간 정책

노출 시작일/종료일이 있다. API 조회 시 서버가 기간을 기준으로 자동 필터링할지, 클라이언트가 받은 뒤 숨길지 정해야 한다. 운영에서는 서버 필터링이 안전하다.

## 스펙화 전에 정해야 할 질문

- 추천 위스키 항목은 단순 배열로 저장할 것인가, 별도 item 테이블로 정규화할 것인가?
- 위스키 상세 정보는 항상 live hydrate할 것인가?
- 큐레이터 코멘트의 최대 길이는 얼마인가?
- 큐레이션 설명과 항목 코멘트에 rich text를 허용할 것인가?
- 노출기간이 지난 큐레이션은 목록 API에서 제외할 것인가?
- 활성화 토글과 노출기간 중 어느 조건이 우선인가?
- 삭제된 alcohol이 포함된 경우 모바일에서 어떻게 보여줄 것인가?

## 도메인 해석

이 이미지는 기존 `CurationKeyword`처럼 위스키 ID만 묶는 기능보다 한 단계 더 풍부하다. 위스키 목록 자체는 alcohol 도메인을 참조하지만, 큐레이션의 본문 질서는 `큐레이터가 어떤 순서로 어떤 코멘트를 붙여 발행했는가`에 있다. 따라서 도메인 중심은 alcohol이 아니라 curation item이다.
