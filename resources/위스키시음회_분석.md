# 위스키시음회 이미지 분석

## 이미지 개요

- 원본 이미지: `위스키시음회.png`
- 화면 성격: 어드민 시음회 큐레이션 작성 화면과 모바일 사용자 노출 화면을 함께 보여주는 시안
- 큐레이션 유형: 오프라인 또는 이벤트성 위스키 시음회 안내 콘텐츠
- 핵심 패턴: 기본 발행 정보 + 시음회 일정/장소 + 참가 정보 + 시음 위스키 목록

## 화면에서 확인되는 구성

### 1. 기본정보 영역

첫 번째 섹션은 공통 발행 메타데이터다.

- 큐레이션명 필수 입력
- 큐레이션내용 필수 입력
- 이미지 업로드 필수
- 노출기간 시작일 필수
- 노출기간 종료일 필수
- 노출순서 입력
- 활성화 여부 토글

이 영역은 다른 큐레이션 유형과 동일하게 재사용할 수 있다.

### 2. 날짜 및 장소 영역

두 번째 섹션은 이벤트 발생 정보를 담는다.

- 시음일 날짜 필수
- 시음회 시간 필수
- 장소 및 네이버 주소 필수
- 검색 또는 주소 검색 버튼
- 상세 주소 필수

모바일 출력에서는 이 정보가 `도시남 X 보틀노트 시음회` 제목 아래의 핵심 안내 카드에 들어간다. 날짜, 시간, 장소, 유의사항 또는 설명이 카드 형태로 묶여 보인다.

### 3. 참가 정보 영역

세 번째 섹션은 참가 조건과 결제성 정보를 담는다.

- 사전 예약/예약 필요 여부로 보이는 토글
- 참가비 입력 필수
- 총 모집 정원 필수
- 신청하기 관련 문구 또는 링크 입력
- 안내사항 입력

모바일 출력 하단에는 `시음회 신청하기` 버튼이 있다. 즉 이 큐레이션은 단순 정보 노출을 넘어 CTA를 포함한다.

### 4. 시음 위스키 영역

네 번째 섹션은 시음회에서 제공되는 위스키 목록이다.

- 섹션 제목: `시음 위스키`
- 반복 단위: `위스키 1`, `위스키 2`, ...
- 각 항목은 위스키 검색/선택과 코멘트 입력을 포함한다.
- 선택된 위스키는 이미지, 이름, 기본 정보, 태그가 포함된 카드로 표시된다.
- 항목 삭제 버튼이 있다.
- 항목 추가 버튼이 있다.

추천큐레이션과 유사하지만, 이 경우 위스키 코멘트는 추천 이유라기보다 시음회 라인업 설명 또는 시음 순서 설명으로 쓰일 가능성이 높다.

## 모바일 출력에서 확인되는 구조

모바일 미리보기는 이벤트 상세 화면 형태다.

- 상단 앱 바
- 커버 이미지 또는 배경 영역
- 이벤트/시음회 유형 배지
- 큐레이션 제목
- 간단한 메타 텍스트
- 소개 문단
- 일정/장소 안내 카드
- 시음 위스키 섹션
- 위스키 카드 목록
- 하단 CTA 버튼

일정/장소 안내 카드는 다음 정보를 포함하는 것으로 보인다.

- 시음 일시
- 장소명
- 주소 또는 위치 설명
- 참가비 또는 신청 관련 안내
- 부가 안내 문구

하단 CTA는 사용자가 시음회 신청 또는 외부 링크 이동을 수행하는 액션이다.

## 출력해야 할 것

이 유형은 콘텐츠 큐레이션이면서 이벤트 도메인 성격을 가진다. 사용자 출력은 단순 상세 정보가 아니라 신청 가능 여부와 CTA를 포함해야 한다.

필수 출력 후보:

- 큐레이션 id
- 큐레이션 유형
- 큐레이션명
- 큐레이션내용
- 커버 이미지 URL
- 노출 시작일
- 노출 종료일
- 노출순서
- 활성화 여부
- 시음일
- 시음 시작 시간
- 시음 종료 시간 또는 시간 텍스트
- 장소명
- 주소
- 상세 주소
- 위도
- 경도
- 참가비
- 총 모집 정원
- 사전 예약 필요 여부
- 신청 CTA 문구
- 신청 CTA URL 또는 액션 타입
- 안내사항 목록
- 시음 위스키 목록

시음 위스키 출력 후보:

- displayOrder
- alcoholId
- hydrate된 alcohol 카드 정보
- tastingComment

위스키 카드 출력 후보:

- alcoholId
- korName
- engName
- imageUrl
- category
- abv
- volume
- region 또는 country
- cask
- averageRating
- tastingTags

## 주의해야 할 것

### 1. 큐레이션과 이벤트의 경계

이 화면은 큐레이션이지만 이벤트 관리 기능의 성격도 강하다. 단순 콘텐츠로 볼지, 실제 신청/정원/상태를 관리하는 이벤트 도메인으로 볼지 먼저 분리해야 한다.

초기에는 `시음회 소개 큐레이션`으로 제한하고, 실제 예약/결제/정원 차감은 별도 도메인으로 미루는 것이 안전하다.

### 2. 날짜 개념이 2종류 존재

기본정보의 노출기간과 시음회 발생일은 다르다.

- 노출기간: 사용자가 콘텐츠를 볼 수 있는 기간
- 시음일/시간: 실제 이벤트가 발생하는 시점

두 날짜를 혼동하면 종료된 시음회가 계속 신청 가능하게 보이거나, 아직 노출 전인 시음회가 검색되는 문제가 생긴다.

### 3. 신청 버튼 정책

모바일 하단에 CTA가 있다. 이 버튼은 외부 링크인지, 내부 신청 플로우인지, 단순 안내 버튼인지 결정해야 한다.

운영에서 외부 신청 링크를 허용한다면 URL 검증, 비활성 상태, 마감 상태, 추적 파라미터 정책이 필요하다.

### 4. 참가비와 정원은 표시 정보인지 운영 데이터인지 결정 필요

참가비와 정원이 단순 표시용이면 payload에 문자열/숫자로 저장 가능하다. 실제 결제나 신청 가능 인원과 연결된다면 별도 이벤트/예약 도메인이 필요하다.

### 5. 주소와 좌표 처리

장소 입력에 주소 검색 버튼이 있다. 주소, 상세주소, 위도, 경도 중 어떤 값이 필수인지 정해야 한다. 지도 연동이 있다면 좌표는 필수에 가깝다.

### 6. 시음 위스키 순서

시음회 라인업은 순서가 의미를 가질 수 있다. 첫 잔, 메인, 마무리처럼 시음 흐름이 있으므로 배열 순서 또는 `displayOrder`를 반드시 보존해야 한다.

### 7. 과거 이벤트 출력

시음일이 지난 경우에도 아카이브로 노출할지, 신청 버튼만 제거할지, 목록에서 제외할지 정책이 필요하다.

### 8. 알코올 카드 정보의 깊이

시음회 화면에서는 위스키 정보가 행사 설명을 보조한다. 추천큐레이션보다 더 짧은 카드가 적합할 수 있다. 리뷰, 찜, 평점까지 모두 보여줄지 신중히 정해야 한다.

## 스펙화 전에 정해야 할 질문

- 이 기능은 단순 시음회 소개 콘텐츠인가, 실제 이벤트 신청 도메인인가?
- 신청 버튼은 외부 링크인가, 내부 신청 API인가?
- 정원은 단순 표시값인가, 신청 수와 연동되는 값인가?
- 참가비는 숫자 원화로 고정할 것인가, 무료/별도문의 같은 문자열을 허용할 것인가?
- 시음 시작/종료 시간을 분리 저장할 것인가, 하나의 텍스트로 저장할 것인가?
- 주소 검색 결과의 좌표가 필수인가?
- 과거 시음회는 앱에서 어떻게 노출할 것인가?
- 시음 위스키는 최소 몇 개, 최대 몇 개까지 허용할 것인가?

## 도메인 해석

이 이미지는 세 이미지 중 가장 리스크가 크다. 추천과 페어링은 콘텐츠 큐레이션에 가깝지만, 시음회는 이벤트, 장소, 일정, 신청 CTA가 결합되어 있다. 따라서 초기 도입 시에는 신청/결제/정원 차감을 포함하지 않는 읽기 전용 이벤트 큐레이션으로 범위를 제한하는 것이 안전하다.
