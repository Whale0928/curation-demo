# 위스키페어링 이미지 분석

## 이미지 개요

- 원본 이미지: `위스키페어링.png`
- 화면 성격: 어드민 큐레이션 작성 화면과 모바일 사용자 노출 화면을 함께 보여주는 시안
- 큐레이션 유형: 위스키와 음식 또는 디저트를 함께 추천하는 페어링 큐레이션
- 핵심 패턴: 기본 발행 정보 + 페어링 조합 N개 + 각 조합별 설명

## 화면에서 확인되는 구성

### 1. 기본정보 영역

첫 번째 섹션은 추천큐레이션 화면과 동일한 공통 발행 정보 영역이다.

- 큐레이션명 필수 입력
- 큐레이션내용 필수 입력
- 이미지 업로드 필수
- 노출기간 시작일 필수
- 노출기간 종료일 필수
- 노출순서 입력
- 활성화 여부 토글

이 영역은 큐레이션 유형과 무관한 공통 헤더로 분리할 수 있다.

### 2. 위스키, 음식 영역

두 번째 섹션은 페어링 조합을 만드는 영역이다.

- 섹션 제목: `위스키, 음식`
- 반복 단위: 페어링 조합
- 각 조합에는 위스키 선택 카드가 있다.
- 각 조합에는 음식 이미지 업로드 영역이 있다.
- 각 조합에는 음식명 입력이 있다.
- 각 조합에는 페어링 설명 입력이 있다.
- 페어링 조합 추가 버튼이 있다.
- 선택된 조합 삭제 버튼이 있다.

첫 번째 조합은 위스키가 선택된 상태이며, 음식 정보 입력 영역이 비어 있다. 하단에는 `+ 페어링 음식 추가`, `+ 페어링 조합 추가`로 보이는 버튼이 있다. 즉 한 페어링 조합 안에서 음식 항목을 늘릴 수 있고, 전체 조합도 늘릴 수 있는 구조로 해석된다.

## 모바일 출력에서 확인되는 구조

모바일 미리보기는 페어링 콘텐츠 상세 화면이다.

- 상단 앱 바
- 커버 이미지 또는 배경 영역
- 큐레이션 유형 배지
- 큐레이션 제목
- 짧은 메타 텍스트
- 큐레이션 설명
- `페어링 큐레이션` 또는 유사 섹션
- 대표 위스키 카드
- 음식 카드 또는 음식 리스트

모바일 화면에서는 위스키 카드가 먼저 나오고, 그 아래에 음식 이미지가 포함된 페어링 설명 카드들이 이어진다. 음식 카드에는 작은 이미지 썸네일, 음식명, 설명 텍스트가 포함된다.

## 출력해야 할 것

이 유형의 출력은 단순 위스키 목록보다 중첩 구조가 필요하다. 하나의 페어링 조합은 위스키와 음식 목록을 함께 가져야 한다.

필수 출력 후보:

- 큐레이션 id
- 큐레이션 유형
- 큐레이션명
- 큐레이션내용
- 커버 이미지 URL
- 노출 시작일
- 노출 종료일
- 노출순서
- 활성화 여부
- 페어링 조합 목록
- 각 조합의 정렬 순서
- 각 조합의 위스키 참조
- 각 조합의 음식 목록

페어링 조합 출력 후보:

- pairingId 또는 배열 index
- alcoholId
- hydrate된 alcohol 카드 정보
- pairingItems
- pairingNote 또는 조합 설명

음식 항목 출력 후보:

- itemName
- itemImageUrl
- itemDescription
- displayOrder

위스키 카드 출력 후보:

- alcoholId
- korName
- engName
- imageUrl
- category
- abv
- volume
- region 또는 country
- cask
- averageRating
- tastingTags

## 주의해야 할 것

### 1. 조합과 음식의 중첩 cardinality

화면상 `페어링 음식 추가`와 `페어링 조합 추가`가 별도로 존재한다. 즉 구조가 다음 중 무엇인지 먼저 확정해야 한다.

- 위스키 1개 + 음식 N개
- 위스키 N개 + 음식 1개
- 위스키 N개 + 음식 N개
- 페어링 조합 N개, 각 조합마다 위스키 1개와 음식 N개

현재 이미지는 `조합 N개, 조합마다 위스키 1개와 음식 N개`에 가장 가깝다.

### 2. 음식은 외부 도메인인지 큐레이션 내부 값인지 결정 필요

위스키는 alcohol 마스터를 참조한다. 반면 음식은 별도 food 마스터가 없어 보이며, 큐레이션 내부 payload로 입력하는 임시 콘텐츠처럼 보인다. 운영에서 음식 검색/재사용이 필요하면 food 도메인으로 분리해야 하지만, 현재 이미지만 보면 curation item 내부 값으로 충분하다.

### 3. 음식 이미지 업로드 정책

음식 이미지는 큐레이션 본문에 종속된다. alcohol 이미지와 달리 마스터 이미지가 아니므로 업로드, 삭제, 재사용, CDN 경로, 대체 이미지 정책이 별도로 필요하다.

### 4. 모바일 출력의 정보 밀도

페어링 화면은 위스키 카드와 음식 카드가 함께 표시된다. 각 카드가 너무 많은 필드를 출력하면 모바일에서 흐름이 무거워진다. 위스키는 핵심 요약만, 음식은 이미지와 설명 중심으로 나누는 것이 적절하다.

### 5. 순서 보존

페어링 조합의 순서와 조합 안 음식의 순서가 모두 의미를 가진다. 배열 순서를 그대로 보존하거나 `displayOrder`를 두 단계로 관리해야 한다.

### 6. hydrate 실패 처리

위스키 hydrate가 실패해도 음식 정보는 남아 있을 수 있다. 전체 페어링 조합을 숨길지, 위스키 영역만 fallback 처리할지 정책이 필요하다.

### 7. 페어링 설명의 소유권

페어링 설명은 alcohol 리뷰도 음식 설명도 아니다. 특정 큐레이션의 특정 조합에 붙는 콘텐츠이므로 `pairingNote` 같은 큐레이션 payload 필드로 보관해야 한다.

## 스펙화 전에 정해야 할 질문

- 하나의 페어링 조합은 위스키를 몇 개까지 가질 수 있는가?
- 하나의 페어링 조합은 음식을 몇 개까지 가질 수 있는가?
- 음식은 이름과 이미지, 설명만 있으면 되는가?
- 음식 이미지가 필수인가?
- 페어링 설명은 조합 단위인가, 음식 항목 단위인가?
- 같은 위스키가 여러 조합에 중복 등장해도 되는가?
- 같은 음식이 여러 조합에 중복 등장해도 되는가?
- 모바일 출력에서 위스키 카드와 음식 카드의 정렬 순서는 고정인가?

## 도메인 해석

이 이미지는 alcohol을 중심으로 한 추천이 아니라 `관계 콘텐츠`를 만드는 기능이다. 핵심 도메인은 위스키도 음식도 아니고, 둘 사이의 페어링 관계다. 따라서 저장 모델은 단순 `alcoholIds` 목록이 아니라 `pairing group` 또는 `pairing item`을 표현할 수 있어야 한다.
